# 数字・主張の事実整合チェック

技術記事の信頼性は、数字と主張の扱い方でほぼ決まる。少しでも「盛っている」と感じられた瞬間に読者の信頼が落ちる。逆に、控えめでも事実通りに書かれていると、内容そのもの以上の説得力が生まれる。

このリファレンスは、書き手と一緒に記事を磨くときに **絶対チェックすべき項目** をまとめる。

---

## チェック観点

### 1. 最悪値だけを切り取って改善幅を強調していないか

**ありがちなパターン:**

ある期間の指標が `132〜316` のレンジで分布していて、改善後（直近）の値が `92` だったとする。それを「指標が 316 → 92 に改善」と書く。

**何が問題か:**
最悪値（316）と新しい値（92）を比較すると、改善幅が実態以上に大きく見える。本文の表でレンジ表記しているのに、タイトルや見出しでは最悪値だけを抜き出すのはダブルスタンダードで釣り感が出る。

**直し方:**
レンジ全体を提示する。

```
❌ 指標が 316 → 92 に改善
✅ 指標が 132〜316 → 92 に改善
```

タイトル・本文・表のすべてが同じ表現で揃うようにする。

---

### 2. 根拠の薄い但し書きを書いていないか

**ありがちなパターン:**

> ただし、改善は施策の前後で滑らかにつながっているので、この施策だけで効いたとは言い切れません。並行して進めた他の取り組みも寄与しているはずです。

**何が問題か:**
「誠実さの装い」になっている。書き手が本当に「他の取り組みも寄与した」と確信しているなら良いが、確信していないのに但し書きを足すと、根拠のない推測を混ぜることになる。読者からすると逆に不誠実に見える。

**直し方:**
書き手に必ず確認する。

> 「この但し書きは本当に思っていますか？ 〇〇が効いたという根拠はありますか？」

書き手が「特に根拠はない」「そう思っていない」と答えたら、段落ごと削除する。事実だけ書く方が信頼性が上がる。

ただし **データから観察できる事実は誠実に書く**。例: 「指標は施策実施よりも前から減少傾向に入っていた」のような、グラフを見れば分かる事実は残す価値がある。

---

### 3. 未確定の予告をしていないか

**ありがちなパターン:**

> 次の打ち手は〇〇です。こちらは別記事で書く予定なので、続報を楽しみにしていてください。

**何が問題か:**
「別記事で書く予定」が確約されていないのに書くと、読者が期待して、後で書けなかったときに信頼を失う。

**直し方:**
確定的な予告を **スタンス表明** に書き換える。

```
❌ 次の打ち手は〇〇です。別記事で書きます。
✅ まだ手をつけられていない領域が残っているので、引き続き取り組んでいく予定です。
```

「やる気はある」「続きはあるはず」という姿勢を示しつつ、確定的な約束はしない。

---

### 4. 専門用語・統計用語の定義が正確か

**ありがちなパターン:**

専門用語の説明で、それっぽいが微妙に間違った定義を書いてしまう。例えば、「上位 10% の重い〇〇がかかる時間」と書いたが、実際は「全体の 90% が〇〇以下に収まる値」が正しい、など。

**何が問題か:**
それっぽく書かれているが意味がボケている、または実は誤りという定義は、詳しい読者から見ると即座に分かる。記事の信頼性に直結する。

**直し方:**
**自分の記憶を信じない**。専門用語、特に統計用語・標準仕様・規格の用語は、必ず一次資料（公式ドキュメント、Wikipedia の最新版、原典の論文など）で定義を確認してから書く。

---

### 5. 引用や出典の年号・固有名詞が正確か

**ありがちなパターン:**

ある概念を「〇〇氏の YYYY 年の発表」と書いたが、実は別の年だった。または、提唱者が別人だった。

**何が問題か:**
こうした誤りは、概念に詳しい読者から見ると即座に分かる。記事全体が「事実確認していない」と判定される。

**直し方:**
書き手と一緒に **WebSearch / WebFetch で必ず確認する**。出典 URL も合わせて貼ると信頼性が増す。

引用が正確かを確認するべきもの:

- 株主への手紙、論文、書籍、講演などの **発行年・著者**
- ライブラリ、フレームワーク、ツールの **正式な綴り**（公式の表記揺れに注意）
- 概念の **提唱者**
- 引用フレーズの **原典での正確な文言**

---

### 6. 集計対象の前提が読者にとって細かすぎないか

**ありがちなパターン:**

> 集計対象は X リポジトリの Y ブランチへの PR で、Z bot は除外しています（13 週間で計 836 PR）。

**何が問題か:**
読者が知りたいのは「どう変わったか」であって、リポジトリ名や bot の除外条件ではない。本題に入る前に細かい前提を並べると読者の負荷が上がる。

**直し方:**
細かい前提は思い切って削るか、脚注に逃がす。後続の表で実数値を出すので、集計の前提は数字を見るときに自然と読者に伝わる。

```
❌ 集計対象は X の Y で、Z は除外しています（13 週間で計 836 PR）。
✅ 運用を開始してから 2 週間が経ちました。導入前と比較して変化が出てきています。
```

ただし **数字を出すときは集計の前提を必ず示す**。最低限「いつから・何件のうち」が分かれば良い。

---

### 7. 見込みと結果の関係が誇張されていないか

**ありがちなパターン:**

> 当初は半分くらい改善できればと思っていたので、想定を超える比率でした。

**何が問題か:**
書き手が実際にそう思っていなかったのに、編集者が「想定を超える」と書くと、書き手の感覚と乖離する。書き手が「2/3 くらいは可能と見込んでいた」と思っていたなら、結果はおおむね想定どおり。

**直し方:**
書き手に確認する。

> 「施策を始める前、どれくらいの改善を見込んでいましたか？」

書き手の実感に合わせる。

```
❌ 想定を超える比率でした。
✅ おおむね想定どおりの比率でした。
```

---

### 8. 比較対象のない数字を出していないか

数字の意味を読者が判断しやすくするため、できれば **比較対象** を添える。

```
❌ 指標は 92 まで下がりました。
✅ 指標は過去 13 週で 132〜316 のレンジだったところ、直近週は 92 まで下がりました。
```

---

## 直すときの作業手順

1. **本文の数字を全部洗い出す**（リード・サブセクション・タイトル・表・まとめ）
2. **同じ数字が複数箇所に出てきた場合、表記が揃っているか確認**
3. **最悪値だけを切り取った数字がないか確認**
4. **但し書きが推測で書かれていないか書き手に確認**
5. **未確定の予告がないか確認**
6. **専門用語の定義が正確か必要なら検索で裏取り**
7. **引用元の年号・固有名詞を必要なら検索で裏取り**

---

## 自分に問うべき質問

「この数字、最悪値を切り取っていないか？」
「この但し書き、書き手は本当にそう思っているか？」
「この予告、書き手は本当に書く予定があるか？」
「この概念、本当に正しい定義か？」
「この引用、本当に正しい出典か？」

これらに即答できないなら、書き手に確認するか一次資料で裏取りする。確認した結果、根拠がないなら削除する。
