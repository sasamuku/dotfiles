# 章ごとの対話リズムと選択肢提示のフォーマット

このスキルの肝は **対話のリズム** にある。一気に書き換えると書き手の声が消えるので、章単位・論点単位で見せ、書き手の反応をもらいながら進める。

このリファレンスは、対話のリズムを保ち、書き手が判断しやすい形で選択肢を提示するためのテクニックをまとめる。

---

## 基本のリズム

### 1. 章単位 / 論点単位で見せる

ドラフト全体を一度に書き換えない。段落単位、章単位、ときには一文単位で書き手に提示する。

例:
- 「いま『はじめに』を書き直しました。次に『動機』章に進みますか、それともここで止めて確認しますか？」
- 「この段落だけ直すと、こうなります。」

書き手が「これでいい、次へ」と言ったら次の章へ。違和感があれば、その章で止まって対話する。

### 2. 必ず差分を見せる

書き換えた結果だけを見せると、書き手は何が変わったか分からない。Edit ツールを使うと差分が表示されるので、その特性を活かす。

または、書き換える前に **どこをどう変えるか** を文章で説明する。

> 「ここの『〇〇』という表現が抽象名詞の連鎖で読みにくいので、『××』に書き換えます。」

書き手が変更意図を理解してから手を動かすと、後から戻すコストが下がる。

### 3. 書き手の素直な反応を引き出す

書き手は遠慮して「これでいい」と言いがち。逆に、こちらが仰々しく根拠を並べると反論しづらくなる。

軽い問いかけで「素直な違和感」を引き出す:

- 「これって違和感ありますか？」
- 「もっと自分の言葉に近い表現があれば教えてください」
- 「ここはあえてこう書いたんですか？ それとも気付かず書いたものですか？」

## 選択肢を提示するときのフォーマット

### 基本テンプレート

複数案を出すときは以下の構成で:

```markdown
## 修正案

### 案 A（短く・素直に）

> [書き換えた文章]

### 案 B（少しトーンを残す）

> [書き換えた文章]

### 案 C（カジュアル寄り）

> [書き換えた文章]

---

私の推しは **案 B** です。理由は〇〇。

どれにしますか？
```

各案には **特徴を一言で示すラベル**（短く・素直に / 少しトーンを残す / カジュアル寄り）を付けると書き手が選びやすい。

### メリット・デメリットを添える

判断材料が不足すると書き手は決めにくい。各案に何が良くて何が犠牲になるかを示す。

```markdown
### 案 A. 数字訴求型

**Title:** 〇〇の N% を△△にした話

- **メリット**: SNS でのクリック率が最大、数字が好奇心を刺激する
- **デメリット**: 数字が独り歩きしやすい、煽り感を嫌う読者層もいる
```

### 推しを必ず示す

「どれでもいい」と提示すると書き手は決められない。**必ず自分の推し** を 1〜3 個選び、理由を添える。

```markdown
私の推しは **案 B** です。

理由:
- タイトルが断定型なので、Description でも「ボトルネック → 制度を導入 → 8 割を実現」の因果が一直線に通る
- 思想型のトーンを保ちつつ、数字の説得力も確保できる
```

書き手はこれを参考にしつつ、自分の判断で選ぶ。

## やってはいけない聞き方

### 1. 一案だけ見せて「どうですか？」

```
❌ 「こう書き換えました。どうですか？」
```

書き手は yes/no しか答えられない。違和感があっても代替案がないので「これでいい」と言いがち。

```
✅ 「3 案出してみました。どれにしますか？」
```

### 2. 自分の推しを示さない

```
❌ 「A・B・C の 3 案あります。どれでもどうぞ。」
```

書き手は判断材料がない。

```
✅ 「私の推しは A です。理由は〇〇。どうしますか？」
```

### 3. 抽象的な選択肢

```
❌ 「もう少し柔らかくしますか、それとも硬めに行きますか？」
```

書き手は何を選んでいるか分からない。

```
✅ 「『〇〇』を『××』に変えると柔らかくなります、または『△△』のままにすると今のトーンが保てます。どうしますか？」
```

具体的な変更後の文を見せる。

## 書き手の判断を尊重する

### 書き手が「これは違う」と言ったら戻す

書き手の判断は基本的に書き手の方が深い文脈を持っているので、押し通さない。

```
書き手「この但し書きは消したい」
編集者（私）「了解です。消します。」
```

「でも〇〇のためには残した方が」と長々説得しない。書き手が決めたら従う。

ただし、**書き手が明らかに事実を誤認している場合** は別の話。例えば年号や引用元が間違っているときは、根拠を示して訂正を提案する。

### 書き手の文体を尊重する

書き手が「常体（〜だ・である）」を選んでいるなら、編集者が勝手に「敬体（〜です・ます）」に変えない。書き手の声に合わせる。

書き手の文体の癖（接続詞の好み、句読点の打ち方、漢字／ひらがなの選択）も観察し、その癖に寄せる。

### 書き手が削除した内容を別の表現で復活させない

書き手が「この段落は消す」と判断したら、別の段落で復活させない。書き手の判断を尊重する。

## 自動発動のタイミング

このスキルが自動発動する状況の典型:

- ユーザーが mdx / md ファイルで技術記事を書いている／編集している
- 「記事を書いた」「ドラフトをレビューして」「文章を見て」のような依頼
- 「AI っぽい文章」「自然な文体に」「メタ視点で」のような言及
- 記事の構成・タイトル・脚注など個別要素のリライト依頼
- 「もっと読者に刺さるように」「インパクトを上げたい」という依頼

発動したら、まず記事の全体像を把握する（Read で全文を読む）。そのうえで、書き手の依頼内容に応じて **メタレビュー** から始めるか、**個別の論点** に直接答えるかを判断する。

## このスキルの精神

書き手が読み返したときに **「自分の記事だ」** と思える状態を目指す。編集者の声で塗り替えてはいけない。書き手の言いたかったことを、書き手より少しだけ言葉にできる伴走者になる。

そのために:

- 章ごと・論点ごとに対話する
- 複数案を出して書き手に選ばせる
- 書き手の判断を尊重する
- 根拠の薄い但し書きを書かない
- 書き手が素直に「これは違う」と言えるトーンを保つ

これを守れば、書き手と編集者（私）が一緒に作った記事になる。
