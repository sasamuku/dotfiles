# Derive Optimal Solution

相手が設定した問題空間の中で解を探すのではなく、**問題を再構成し、選択肢を棚卸しし、評価し、最適解を推奨する** 汎用思考プロトコル。ドメイン非依存。

## 典型的な失敗

相手が「A という問題があるので B という対策を考えている」と聞いてくる → A を所与として B の実装詳細に踏み込む → A を発生させている上流 C を見逃す (C を変えれば A も B も不要になる可能性)。

**問題は定義ではなく解釈の対象**として扱う。

## 発動条件 (いずれか該当で必ず適用)

1. 評価・再考を求める語:「筋が良い」「ほんとうに」「そもそも」「最善は」「根本は」「本当に必要か」
2. 判断 / 選択 / 方針 / 戦略 / プロセスの議論
3. 相手から新しい具体案を受けた (妥当性を多視点で検証)
4. 問題が複数レイヤー (技術・プロセス・組織・規制・UX・コスト等) をまたぐ
5. 自分が出した解決策を相手が保留・疑っている
6. トレードオフが明確にある

問題のサイズに応じて深さを調整:
- 小さな相談: Step 1〜2 のみ軽く
- 中規模: 全ステップ通すが各 1〜2 案
- 大きな判断: 厳密に全ステップ、5〜7 案、採点表、不確実性明記

## 思考プロトコル (6 ステップ)

解決策を書く前に必ず通す。スキップ項目があれば解決策提示は保留。

**ステップの対応関係**: Step 1〜2 は出力の「問題の再構成」節に統合、Step 3 は「選択肢の棚卸し」の材料、Step 4 は「選択肢の棚卸し」、Step 5 は「評価軸」、Step 6 は「採点表」にそれぞれ対応する。

**Step 3 (問題空間の拡張) vs Step 5 (評価軸) の境界**: Step 3 は **問題の見方を変える** (「この問題は誰のための問題か」「境界はどこか」)。Step 5 は **生成した案の良し悪しを測る** (「この案は根本に届くか」「可逆か」)。

迷ったら以下の対応例に従う:

| 概念 | Step 3 (問題の性質) での扱い | Step 5 (案の評価) での扱い |
|---|---|---|
| 可逆性 | 「この問題は今やらないと不可逆か、後でも間に合うか」(時間軸 / 前提条件) | 「この案は失敗したときに戻せるか」(評価軸の可逆性) |
| 主体 | 「誰のための問題か」で問いを変える (主体) | 扱わない |
| コスト | 「所与の予算制約は本当か」(前提条件) | 「この案の実行・維持コストはどうか」(評価軸のコスト) |
| 前例 | 「類似事例でどう解かれたか」(業界・前例) | 扱わない |

同じ語が両方に登場したら、Step 3 は **問題の性質**、Step 5 は **案の性質** と覚える。

### Step 1: 症状 vs 構造を切り分ける

相手は「症状」を話している可能性がある。症状 ≠ 問題。

- 症状例:「エラーが出る」「会議が長い」「チームが疲弊」「売上が伸びない」
- 構造例:「権限設計が破綻」「意思決定の所有者不在」「目標とインセンティブが逆行」「顧客セグメントと提供価値が噛み合わない」

**書き下す**: 相手の問題 X は症状か、構造か。

### Step 2: 上流に 2 段遡る

「なぜ」を 2 回以上。1 回で止まらない。

- なぜ発生する? → 直接原因
- なぜ直接原因がある? → **上流の要因 / 設計 / 前提**
- 上流を変えれば問題が消える可能性があるか?

**書き下す**: 上流の要因 X は何か。X を変える選択肢は存在するか。

### Step 3: 問題空間の境界を広げる (6 次元)

相手の問題空間の**外側**に候補があるか。各次元で問い直す:

| 次元 | 問い |
|---|---|
| 時間軸 | 短期 / 中期 / 長期で答えが変わるか |
| 主体 | 誰が / 誰のために / 誰と解くか |
| スコープ境界 | 境界を引き直せば問題が消えないか |
| 前提条件 | 所与の制約は本当に必要か (規制・予算・技術・関係者) |
| 抽象レベル | 症状 / 構造 / 原理のどこで解くか |
| 業界・前例 | 既知の解・類似事例 (業界 / 同種プロダクト / 自組織) |

(可逆性・漸進性は Step 5 の評価軸で扱う。Step 3 は問題の見方を変える)

非技術文脈の例:
- 時間軸: 「今年の売上問題」を「3 年後の市場位置取り」と解釈し直すと答えが変わる
- 主体: 「エンジニアの負荷問題」を「採用市場での雇用主競争力」として解くと選択肢が広がる
- 前提条件: 「Slack で決めるべき」と言われた議論を「そもそも非同期でいいか」から問う

### Step 4: 案を 3 つ以上生成する

最低 3 案、できれば 5〜7 案。比較軸が生まれる。

**「枠組みの外側にある選択肢」の判定テスト**: 候補案を「相手の提案と並べたとき、以下のどれか 1 つが異なるか」で判定する。どれも同じなら内側、1 つでも異なれば外側。

1. **解く場所** (どの層 / どの制御点で解くか) が違うか
2. **抽象レベル** (症状対処 / 構造変更 / 原理の再定義) が違うか
3. **時間軸** (今実行する / 先に検証する / 延期 / やらない) が違うか

**「解く場所」は層 + 制御点の 2 段で捉える**:

- **層**: どの責務領域か (ドメイン固有。技術なら UI / アプリ / DB / インフラ、非技術なら個人 / チーム / 組織 / 契約など)
- **制御点**: その層の中で、どの要素を操作するか (構造自体を変える / 制約を変える / 入力や前提を絞る / 処理を変える / 境界を引き直す など)

同じ層に見えても制御点が違えば別の解。特に **「構造を新しく作る」より「既存の入力・前提を絞る」** ほうが大抵コストが低い — この選択肢を見落としやすいので、層を決めたら必ず「この層の中で、触れる制御点を全部挙げたか?」を自問する。

単なる「相手が思いつかなかった具体案」(= 3 軸すべて同じだが別アイデア) は **内側**。

**Step 3 (スコープ境界) との違い**: Step 3 は「問題の見方を変える」(「誰の問題か」「境界をどこに引くか」)、Step 4 の外側は「解の出し場所を変える」(「どこで・どの抽象で・いつ解くか」)。Step 3 は問題側、Step 4 は解側。

- **「どこで問題を解くか」** を変えて多様化 (原理 / 構造 / 運用 / 契約 / 人事 / 外部依頼 / やらない)
- 相手の案があれば**それを発展させた変形案**を含める
- **却下された案の改善版**がないか検討 (状況が変わっている可能性)
- **「そもそもやらない / 放置 / 延期」**を必ず入れる

### Step 4.5: 解の「重さ」で優先順位をつける

Step 4 で生成した案を、**加える責務の重さ** で並べる。軽いほうから優先して検討する。以下は **ドメイン非依存の 5 段階**:

**軽い順**:
1. **既存の要素に状態を足す** — 既存の何かに属性を 1 つ追加し、既存の判定条件の書き方を変えるだけで済む
2. **既存の入力や前提を絞る** — 新しい仕組みは作らず、既存の処理・運用に渡す入力や適用範囲を限定する
3. **既存の要素に情報を複製する** — 値の複製・コピー保持。複製元が変わったときの同期責務が発生する
4. **新しい要素や仕組みを足す** — 新エンティティ / 新テーブル / 新フロー / 新ルール / 新役割を追加する
5. **既存の原則や境界に例外を足す** — 既存のポリシー・認可・ルール・合意に文脈別の分岐や特例を入れる (多段防御や一貫性を損なうリスク)

ドメイン別の読み替え例:
- **技術**: 1 = 列 / フラグ追加、2 = クエリ条件・スコープ・セッション変数を絞る、3 = 非正規化列・キャッシュ、4 = 新テーブル・新サービス、5 = 認可ポリシーに例外分岐
- **プロセス / 運用**: 1 = 既存フォームに項目追加、2 = 既存手順の入力条件を絞る、3 = 別所に情報を複写して保持、4 = 新フロー / 新役職 / 新会議体、5 = 既存ルールに例外運用
- **組織 / 意思決定**: 1 = 既存メンバーに属性を付与、2 = 既存会議体の参加範囲を絞る、3 = 既存文書に別情報を併記、4 = 新委員会 / 新ポジション、5 = 既存の意思決定原則に例外

**原則**: 同等の効果が得られるなら、**重さが軽い案を主推奨にする**。重い案はバックアップ・将来選択肢・部分補強として扱う。

重さの違いは以下で効いてくる:
- **影響範囲**: 新しい仕組みの追加は全体への回帰を引き起こす、既存への状態追加は局所
- **移行コスト**: 情報の複製は過去分への遡及が要る、状態追加はデフォルト値で済む
- **運用責務**: 新フローは人手の追加、既存の絞り込みは場所が明確
- **可逆性**: 新しい仕組みは物理的に残る、状態や条件は書き換えで戻せる

推奨を書くとき、**「なぜ重い案を選ばないか」を明記** する。主推奨が重い案 (重さ 3 以上) になる場合は、軽い案では解けない具体的な制約が何かを示すこと。示せないなら重い案を主推奨にしない。

### Step 5: 評価軸を明示する

「筋が良い」を曖昧にしない。文脈に応じて 3〜6 軸選ぶ:

- **根本性**: 症状ではなく原因に届くか
- **堅牢性**: 抜け漏れ・不確実性・人の入れ替わりに強いか
- **耐久性**: 時間経過・環境変化・要件拡大に耐えるか
- **コスト**: 実行・維持・学習・移行の総量
- **可逆性**: 失敗時に戻せるか
- **明快性**: 当事者外・新参者から理解できるか
- **多段防御**: 1 層破れて別が止まるか (安全重視文脈)

軸選択の指針: 安全性重視なら堅牢性 + 多段防御、スピード重視なら可逆性 + コスト、長期運用なら耐久性 + 明快性。

### Step 6: 採点表で比較する

軸 × 案の表。✅ / ⚠️ / ❌ の 3 段階で十分。

- **相手の提案を必ず 1 列含める** (自分の推しを通すための表にしない)
- 表に落ちない価値 (美学、一貫性、政治的制約) は補足テキストで
- **集計スコア (✅=2, ⚠️=1, ❌=0 等の合計) を作らない**。軸の重要度は文脈依存で一律ではなく、単純合計は質的判断を歪める。採点表はパターンを可視化する道具であって点数で順位をつける道具ではない

## アンチパターン

- **即座に賛成・反対しない**: 相手の具体提案に対して 6 ステップを通さず返答しない
- **自説を防衛しない**: 自分の案 A に相手の案 B が来たら、並べて比較する。自説を捨てる覚悟を持つ
- **1 案で答えない**: 必ず比較可能な 3 案以上。最終推奨は 1 つで良い
- **ドメインに閉じない**: 技術相談でもプロセス・組織・契約・外部委託・やらない、も選択肢
- **前提を所与にしない**:「規制が」「上司が」「慣例で」と言われても、今も有効かを問う
- **却下案を無批判に受け入れない**: 却下理由が弱ければ再浮上させる。却下時と状況が変わっている可能性がある
- **実行詳細に先走らない**: 「どう実行するか」の前に「何を実行するか」「そもそも実行するか」

## 典型的な視点シフト

| 相手が設定した問題 | 上流の問題 |
|---|---|
| API が遅い | データモデルが深い 1:N:N |
| 権限チェックが漏れる | 認可と取得が同じクエリに混在 |
| テストが壊れる | 依存方向が逆転 |
| バグ修正が連鎖する | 責務分割がモジュール境界と不一致 |
| 会議が長い | 議題と決定者が事前未定 |
| チームが疲弊 | 目標とインセンティブが矛盾 |
| リリース事故が続く | 本番と開発の環境差がコードで吸収 |
| 採用がうまくいかない | 基準とオンボーディングが別人格 |
| 売上が伸びない | 顧客セグメントと提供価値が不整合 |
| 方針がぶれる | 決定者と責任者が分離 |

**パターン**: 症状は末端で観測、原因は上流の選択・整合にある。

- 案 A (相手の提案): ...

## 境界判断の原則

本スキルの指示の境界で迷った場合は以下の原則で進む:

- **粒度・階層の選び方** (「解く場所」をアプリ層内でどこまで区別するか等): **その文脈で意味のある最小単位**で切る。技術なら「層 (UI / ドメイン / DB)」、組織なら「役割 (個人 / チーム / 組織)」、時間軸なら「意思決定サイクル」が基本単位
- **情報が足りない次元** (「業界・前例」でシナリオに固有情報がない等): **省略可**。該当なしと明示して進む。無理に埋めない
- **指示同士の競合** (「やらない案を必ず入れる」vs「軸の制約で評価不能」等): **実用性を優先**。形式を満たすために無意味な案を入れない
- **自分で生成した案を自分で却下する構図** (相手の却下案の再浮上ではない): 通常の却下理由で済ませる。「再浮上」アンチパターンは相手の既却下案にのみ適用

迷いは出力に「不明瞭点」として上げる代わりに、**境界判断の根拠を「裁量補完」に書いて進む**。

## チェックリスト (解決策提示の直前)

- [ ] 症状と構造を切り分けたか
- [ ] 上流に 2 段遡ったか
- [ ] 6 次元のうち最低 3 つで境界を広げたか
- [ ] 案を 3 つ以上、相手の提案を 1 列含めたか
- [ ] ドメイン外と「やらない / 延期」を入れたか
- [ ] 評価軸を明示し採点表を作ったか
- [ ] 却下理由と不確実性を書いたか

1 つでも "いいえ" があれば保留して戻る。

## 適用の型 (問題の再構成までの例)

具体の選択肢や推奨は書かない。**Step 1〜2 でどこまで遡れば上流に届くか** の相場感だけ示す。選択肢生成は Step 3〜4 で読者が自分で行う。

### 技術設計 — 参照先の削除で整合性が壊れるパターン

- 設定された問題: 参照先 (ユーザー / カテゴリ / 部署 等) を削除した結果、他所の表示や API が壊れる
- 上流: 削除の粒度 (物理 / 論理 / 匿名化) と参照整合性の要求がすり合っていない
- 本質的な問い: 削除の業務意味と、DB 操作としての削除は同じ操作でよいか

### プロセス — 末端で事故が続くパターン

- 設定された問題: 特定のプロセス段階 (レビュー / リリース / 受付 等) で事故が続く → その段階を強化したい
- 上流: 事故を生む構造 (環境差 / 責任分散 / 手戻り経路) が上流に散在している
- 本質的な問い: 見逃しを減らすことと、見逃しても平気な構造を作ることは別問題ではないか

### 意思決定 — 二者択一で意見が割れるパターン

- 設定された問題: 選択肢 X / Y のどちらを採るか
- 上流: 「何を基準に選ぶか」が合意されていない
- 本質的な問い: 選ぶ前に、選ぶための基準を作る必要はないか

ドメインに関係なく同じプロトコルを適用できる。技術用語で書かれた例でも、枠組みは非技術問題にそのまま転用できる。

## ultrathink との併用

`ultrathink` が「思考時間を使う許可」なのに対し、このスキルは「どこに時間を使うか」の方向付け。併用で**上流の構造を深く掘る**ことに時間を投入できる。
