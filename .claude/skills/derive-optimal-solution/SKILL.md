# Derive Optimal Solution

相手が設定した問題空間の中で解を探すのではなく、**問題を再構成し、選択肢を棚卸しし、評価し、最適解を推奨する** 汎用思考プロトコル。ドメイン非依存。

## 典型的な失敗

相手が「A という問題があるので B という対策を考えている」と聞いてくる → A を所与として B の実装詳細に踏み込む → A を発生させている上流 C を見逃す (C を変えれば A も B も不要になる可能性)。

**問題は定義ではなく解釈の対象**として扱う。

## 発動条件 (いずれか該当で必ず適用)

1. 評価・再考を求める語:「筋が良い」「ほんとうに」「そもそも」「最善は」「根本は」「本当に必要か」
2. 判断 / 選択 / 方針 / 戦略 / プロセスの議論
3. 相手から新しい具体案を受けた (妥当性を多視点で検証)
4. 問題が複数レイヤー (技術・プロセス・組織・規制・UX・コスト等) をまたぐ
5. 自分が出した解決策を相手が保留・疑っている
6. トレードオフが明確にある

問題のサイズに応じて深さを調整:
- 小さな相談: Step 1〜2 のみ軽く
- 中規模: 全ステップ通すが各 1〜2 案
- 大きな判断: 厳密に全ステップ、5〜7 案、採点表、不確実性明記

## 思考プロトコル (6 ステップ)

解決策を書く前に必ず通す。スキップ項目があれば解決策提示は保留。

### Step 1: 症状 vs 構造を切り分ける

相手は「症状」を話している可能性がある。症状 ≠ 問題。

- 症状例:「エラーが出る」「会議が長い」「チームが疲弊」「売上が伸びない」
- 構造例:「権限設計が破綻」「意思決定の所有者不在」「目標とインセンティブが逆行」「顧客セグメントと提供価値が噛み合わない」

**書き下す**: 相手の問題 X は症状か、構造か。

### Step 2: 上流に 2 段遡る

「なぜ」を 2 回以上。1 回で止まらない。

- なぜ発生する? → 直接原因
- なぜ直接原因がある? → **上流の要因 / 設計 / 前提**
- 上流を変えれば問題が消える可能性があるか?

**書き下す**: 上流の要因 X は何か。X を変える選択肢は存在するか。

### Step 3: 問題空間の境界を広げる (6 次元)

相手の問題空間の**外側**に候補があるか。各次元で問い直す:

| 次元 | 問い |
|---|---|
| 時間軸 | 短期 / 中期 / 長期で答えが変わるか |
| 主体 | 誰が / 誰のために / 誰と解くか |
| スコープ境界 | 境界を引き直せば問題が消えないか |
| 前提条件 | 所与の制約は本当に必要か (規制・予算・技術・関係者) |
| 抽象レベル | 症状 / 構造 / 原理のどこで解くか |
| 業界・前例 | 既知の解・類似事例 (業界 / 同種プロダクト / 自組織) |

(可逆性・漸進性は Step 5 の評価軸で扱う。Step 3 は問題の見方を変える)

非技術文脈の例:
- 時間軸: 「今年の売上問題」を「3 年後の市場位置取り」と解釈し直すと答えが変わる
- 主体: 「エンジニアの負荷問題」を「採用市場での雇用主競争力」として解くと選択肢が広がる
- 前提条件: 「Slack で決めるべき」と言われた議論を「そもそも非同期でいいか」から問う

### Step 4: 案を 3 つ以上生成する

最低 3 案、できれば 5〜7 案。比較軸が生まれる。

- **「どこで問題を解くか」** を変えて多様化 (原理 / 構造 / 運用 / 契約 / 人事 / 外部依頼 / やらない)
- 相手の案があれば**それを発展させた変形案**を含める
- **却下された案の改善版**がないか検討 (状況が変わっている可能性)
- **「そもそもやらない / 放置 / 延期」**を必ず入れる

### Step 5: 評価軸を明示する

「筋が良い」を曖昧にしない。文脈に応じて 3〜6 軸選ぶ:

- **根本性**: 症状ではなく原因に届くか
- **堅牢性**: 抜け漏れ・不確実性・人の入れ替わりに強いか
- **耐久性**: 時間経過・環境変化・要件拡大に耐えるか
- **コスト**: 実行・維持・学習・移行の総量
- **可逆性**: 失敗時に戻せるか
- **明快性**: 当事者外・新参者から理解できるか
- **多段防御**: 1 層破れて別が止まるか (安全重視文脈)

軸選択の指針: 安全性重視なら堅牢性 + 多段防御、スピード重視なら可逆性 + コスト、長期運用なら耐久性 + 明快性。

### Step 6: 採点表で比較する

軸 × 案の表。✅ / ⚠️ / ❌ の 3 段階で十分。

- **相手の提案を必ず 1 列含める** (自分の推しを通すための表にしない)
- 表に落ちない価値 (美学、一貫性、政治的制約) は補足テキストで

## アンチパターン

- **即座に賛成・反対しない**: 相手の具体提案に対して 6 ステップを通さず返答しない
- **自説を防衛しない**: 自分の案 A に相手の案 B が来たら、並べて比較する。自説を捨てる覚悟を持つ
- **1 案で答えない**: 必ず比較可能な 3 案以上。最終推奨は 1 つで良い
- **ドメインに閉じない**: 技術相談でもプロセス・組織・契約・外部委託・やらない、も選択肢
- **前提を所与にしない**:「規制が」「上司が」「慣例で」と言われても、今も有効かを問う
- **却下案を無批判に受け入れない**: 却下理由が弱ければ再浮上させる。却下時と状況が変わっている可能性がある
- **実行詳細に先走らない**: 「どう実行するか」の前に「何を実行するか」「そもそも実行するか」

## 典型的な視点シフト

| 相手が設定した問題 | 上流の問題 |
|---|---|
| API が遅い | データモデルが深い 1:N:N |
| 権限チェックが漏れる | 認可と取得が同じクエリに混在 |
| テストが壊れる | 依存方向が逆転 |
| バグ修正が連鎖する | 責務分割がモジュール境界と不一致 |
| 会議が長い | 議題と決定者が事前未定 |
| チームが疲弊 | 目標とインセンティブが矛盾 |
| リリース事故が続く | 本番と開発の環境差がコードで吸収 |
| 採用がうまくいかない | 基準とオンボーディングが別人格 |
| 売上が伸びない | 顧客セグメントと提供価値が不整合 |
| 方針がぶれる | 決定者と責任者が分離 |

**パターン**: 症状は末端で観測、原因は上流の選択・整合にある。

## 出力テンプレート

```markdown
## 問題の再構成
- 相手が設定した問題: ...
- 観測している症状: ...
- **上流の構造**: ...
- **本質的な問い**: ...

## 選択肢の棚卸し
- 案 A (相手の提案): ...
- 案 B (現状維持): ...
- 案 C (上流を変える): ...
- 案 D (既知事例に倣う): ...
- 案 E (前提を崩す): ...
- 案 F (やらない / 延期): ...

## 評価軸
(文脈に応じた 3〜6 軸)

## 採点表
| 軸 | A | B | C | D | E | F |
|---|---|---|---|---|---|---|
| ... |

## 推奨
1 案を選ぶ。選ばなかった案の却下理由も明記。
複数案のシーケンシャル (先に X、結果次第で Y) / 段階的 (Phase 1: X, Phase 2: Y) 組み合わせも許容。

## 不確実性
確信を持てない部分を正直に。
```

## チェックリスト (解決策提示の直前)

- [ ] 症状と構造を切り分けたか
- [ ] 上流に 2 段遡ったか
- [ ] 6 次元のうち最低 3 つで境界を広げたか
- [ ] 案を 3 つ以上、相手の提案を 1 列含めたか
- [ ] ドメイン外と「やらない / 延期」を入れたか
- [ ] 評価軸を明示し採点表を作ったか
- [ ] 却下理由と不確実性を書いたか

1 つでも "いいえ" があれば保留して戻る。

## 適用例

### 技術設計 (DB スキーマ)

- **設定された問題**: タスク担当者の `user: null` 混入 → アプリ層で合成したい
- **上流**: 直接原因は `users` が RLS で不可視、上流は所属テーブルの物理削除
- **本質的な問い**: 業務データを残しつつ本人アクセス遮断する方法として、物理削除が唯一か
- **選択肢**: A (アプリ層合成) / B (型ブランド) / C (ORM upgrade) / **D (論理削除 + context フィルタ) ← 上流を変える**

### プロセス (リリース事故)

- **設定された問題**: 本番事故が続く → レビューを厚くしたい
- **上流**: 直接原因はレビュー見逃し、上流は本番と開発の環境差がコードに散在
- **本質的な問い**: 見逃しを減らすことと、見逃しても大丈夫な構造を作ることは別問題
- **選択肢**: A (レビュー厚く / 症状) / B (CI を厚く) / C (環境差を config 集約 / 構造) / D (feature flag 段階公開 / 被害軽減) / E (カナリアデプロイ) / F (リリース頻度を下げる)

### 意思決定 (採用)

- **設定された問題**: 応募者 X を採用すべきか
- **上流**: 直接原因は評価が割れている、上流は「何ができる人が必要か」の合意不在
- **本質的な問い**: 採る / 採らないの前に採用基準を言語化できているか
- **選択肢**: A (採用) / B (不採用) / **C (基準を作り直してから再判断)** / D (契約で短期試用) / E (別ポジションで再提案) / F (採用枠そのものを見直す)

ドメインに関係なく同じプロトコルを適用できる。技術用語で書かれた例でも、枠組みは非技術問題にそのまま転用できる。

## ultrathink との併用

`ultrathink` が「思考時間を使う許可」なのに対し、このスキルは「どこに時間を使うか」の方向付け。併用で**上流の構造を深く掘る**ことに時間を投入できる。
